# 원격 직접 접수 도입 계획서

---

## 1. 프로젝트 개요


**As-Is**
- 담당 선생님이나 고객센터에 연락하여 원격 접수 요청 필수
- 선생님이나 고객센터 연결이 되지 않을 경우 접수 어려움
- 정보 확인 등의 절차로 고객 불편 가중

**To-Be**
- 기기에서 원격 접수 버튼 터치 시 카카오 알림톡으로 접수 URL 자동 발송
- 학부모가 휴대폰에서 URL 진입 → 직접 원격 접수 가능, 편의성 증대
- 접수 완료 후 알림톡 재발송으로 예약 내역 확인

---

## 2. 기술 스택

별도 프레임워크 없이 순수 HTML/CSS/JavaScript로 구성, 어느 환경에서도 의존성 없이 동작

| 분류 | 기술 |
|------|------|
| 마크업 | HTML5 |
| 스타일 | CSS3 (Vanilla, CSS Custom Properties) |
| 스크립트 | JavaScript ES6+ (Vanilla, Fetch API) |
| 서버 | PHP 8.0+ |
| 보안 | CSRF 토큰, 토큰 기반 인증 (매직 링크) |
| 외부 API | Nager.Date — 한국 공휴일 (서버 사이드, 30일 캐시) |
| 알림 | 카카오 알림톡 — 접수 URL 발송 (CRM/기기 앱 측 처리) |

## 3. 진입 방식 및 인증

학부모 휴대폰에서만 동작, 모바일 전용 레이아웃으로 구성

**진입 흐름**

```
기기(탭/앱)에서 원격 접수 버튼 터치
 → 서버가 1회용 인증 토큰 생성
 → 카카오 알림톡으로 URL 발송 (안내 문구 + 접수 URL 포함)
    예) https://도메인/remote/kids?token=a3f9c2e1...
 → 학부모가 URL 터치 → 토큰 검증 → 회원 정보 로드 → 접수 폼 바로 표시
```

**토큰 방식 (매직 링크)**

| 항목 | 내용 |
|------|------|
| 토큰 형태 | 서버 생성 랜덤 문자열 (계정 정보 미포함) |
| 만료 시간 | 발송 후 30분~1시간 |
| 1회용 처리 | 접수 완료 또는 만료 시 토큰 무효화 |
| URL 노출 정보 | 서비스 키 + 토큰만 포함, 개인정보 미노출 |
| HTTPS | 필수 (토큰이 인증 수단이므로 암호화 필요) |

**배포 서버**

```
isherpa.gscdn.com (외부망 서버)
├── cust_center/launcher/   기존 런처 설치 안내 (그대로 유지)
└── remote/                 원격 접수 프로젝트 (신규)
    ├── kids
    ├── elementary
    ├── middle
    └── high
```

고객(학부모) 휴대폰에서 알림톡 URL로 접근하므로 외부망 서버 필수
기존 런처 프로젝트와 동일 서버 공존, 경로만 분리

**서버 담당자 확인 필요 항목**

| 항목 | 내용 |
|------|------|
| PHP 버전 | 8.0 이상 필요 |
| mod_rewrite | Apache `.htaccess` 동작 여부 |
| MySQL 접근 권한 | `remote_token` DB 생성 권한 |
| 디렉토리 쓰기 권한 | `cache/` 폴더 write 권한 (공휴일 캐시용) |

**DB 구조**

접수 데이터는 CRM DB(SP 경유)로 처리, 별도 자체 DB 불필요
토큰 저장용 테이블 1개만 필요

- 1순위: CRM DB에 테이블 추가 요청
- 2순위: 외부망 서버에 `remote_token` DB 별도 생성

```
remote_token.remote_auth_token 테이블
├── token        랜덤 문자열 (인증 키)
├── user_id      회원 ID
├── service_key  서비스 구분 (kids / elementary 등)
├── expires_at   만료 일시
├── used         사용 여부 (0/1)
└── created_at   발급 일시 (자동 입력)
```

**CRM 연동 구조**

```
학부모 휴대폰
 → isherpa.gscdn.com/remote/ (외부망 서버)
    ├── 접수 등록  → CRM remote_request_insert SP
    └── 슬롯 조회  → CRM 슬롯 SP (명세 미수령 ❌)

CRM DB ←──────── CTI (자동 연동)
                  상담사 상태 변경 → CRM 자동 반영
```

## 4. 파일 구성 및 역할

### 4-1. 서비스별 진입 파일

접수 대상 서비스 4종, 각각 별도 진입 파일로 구분
URL만 다를 뿐 화면 구성·동작 동일, 서비스별 색상 테마만 상이

| 서비스명 | 진입 URL | 파일 | 색상 테마 | 고객센터 |
|---------|----------|------|----------|---------|
| 밀크T아이 | `/remote/kids` | `public/remote/kids.php` | 오렌지 계열 | 1522-6002 |
| 밀크T초등 | `/remote/elementary` | `public/remote/elementary.php` | 블루 계열 | 1577-1533 |
| 밀크T중학 | `/remote/middle` | `public/remote/middle.php` | 그린 계열 | 1522-5533 |
| 밀크T고등 | `/remote/high` | `public/remote/high.php` | 남색/보라 계열 | 1522-5533 |

각 진입 파일 역할:
1. 서비스 키·서비스명·고객센터 번호 정의
2. 토큰 검증 및 회원 정보 로드 (토큰 없거나 만료 시 오류 페이지로 이동)
3. 접수 폼(`form.php`) 직접 호출

### 4-2. 화면 파일 (사용자에게 보이는 페이지)

```
public/pages/
├── form.php        접수 폼 화면 — 연락처·증상·예약일시 입력
├── complete.php    접수 완료 화면 — 예약일시·연락처 확인
└── error.php       공통 오류 화면 — 오류 유형별 메시지 표시  ← 신규
```

| 파일 | 역할 |
|------|------|
| `form.php` | URL 진입 시 바로 표시되는 접수 정보 입력 화면. 토큰 인증 후 세션으로 서비스 정보 사용 |
| `complete.php` | 접수 완료 후 예약일시·연락처 확인 화면. 1회성 세션으로 전달, 새로고침 시 중복 표시 방지 |
| `error.php` | 오류 유형별 메시지만 표시. 토큰 만료, 잘못된 접근, 서버 오류 등 상황별 안내 |

### 4-3. API 파일 (화면에서 비동기 호출)

```
public/api/
├── slots.php       예약 가능한 시간 슬롯 목록 반환
├── submit.php      접수 폼 데이터 처리 및 접수 등록
```

| 파일 | 역할 |
|------|------|

| `holidays.php` | 달력 로드 시 공휴일 날짜 목록 JSON 반환. Nager.Date API 자동 호출 + 30일 캐시. 임시 휴일은 `config/extra_holidays.php`에서 수동 관리 |

## 7. 페이지별 UI 구성

### 페이지 1 — 접수 폼 🔧

알림톡 URL 터치 시 바로 표시
연락처·문의내용·예약날짜·예약시간을 5단계로 나눠 입력, 마지막 단계에서 확인 후 접수

**5단계 스텝 위저드 구조**

```
┌──────────────────────────────────────────┐
│  헤더 (서비스명 + 캐릭터)                 │  서비스별 색상 테마
├──────────────────────────────────────────┤
│  ① 연락처  ② 문의내용  ③ 날짜  ④ 시간대  ⑤ 확인  │  스텝 인디케이터
├──────────────────────────────────────────┤
│  [Step 1]  연락처 입력                    │
│  [ 010-0000-0000 ]                       │
│                                          │
│  [ 다음 →  ]                             │
└──────────────────────────────────────────┘

┌──────────────────────────────────────────┐
│  [Step 2]  문의내용 입력                  │
│  [                                     ] │  최대 2000자 / 글자 수 표시
│                                          │
│  [ ← 이전 ]  [ 다음 → ]                  │
└──────────────────────────────────────────┘

┌──────────────────────────────────────────┐
│  [Step 3]  날짜 선택                      │
│  [ 날짜 카드 그리드 (오늘~6일 후) ]        │
│                                          │
│  [ ← 이전 ]  [ 다음 → ]                  │
└──────────────────────────────────────────┘

┌──────────────────────────────────────────┐
│  [Step 4]  시간대 선택                    │
│  [ 13:00  14:00  15:00  16:00 ... ]      │  슬롯 목록
│                                          │
│  [ ← 이전 ]  [ 다음 → ]                  │
└──────────────────────────────────────────┘

┌──────────────────────────────────────────┐
│  [Step 5]  접수내용 확인                  │
│  ──────────────────────────────────────  │
│  연락처    010-1234-5678                 │
│  문의내용  화면이 켜지지 않아요           │
│  예약일시  3월 25일(수) 13:00            │
│  ──────────────────────────────────────  │
│  [ ← 이전 ]        [ 접수하기 ]          │
└──────────────────────────────────────────┘
```

**구현 포인트**
- 연락처 입력 시 하이픈 자동 삽입 (010-0000-0000)
- 문의내용 글자 수 실시간 카운터, 최대치 근접 시 색상 경고
- 예약 가능 날짜: 오늘부터 6일 후까지 (오늘 포함 총 7일, 예: 월요일 접수 시 일요일까지)
- 예약 가능 시간: 13:00~22:00, 1시간 단위 (17:00~18:00 제외)
- 날짜 선택 시 시간 슬롯 백그라운드 프리로드
- 각 스텝마다 클라이언트 유효성 검사 후 다음 단계 이동
- 서버 유효성 검사는 최종 접수 시 수행
- 접수 시 중복 여부 확인 후 아래 분기 처리

**시간 슬롯 표시 규칙**

| 조건 | 처리 방식 |
|------|---------|
| 17:00~18:00 (식사 시간) | 슬롯 목록에서 미표시 |
| 당일 접수 시 현재 시간대 | 슬롯 목록에서 미표시 (45분 마감 기준으로 단순화) |
| 당일 접수 시 이미 지난 시간대 | 슬롯 목록에서 미표시 |
| 익일 이후 접수 | 전체 시간 슬롯 표시 (17:00~18:00 제외) |

- 비활성(회색) 처리가 아닌 **목록에서 완전히 제거**

**중복 접수 분기**

| 상황 | 안내 문구 | 버튼 |
|------|---------|------|
| 종료되지 않은 접수 건이 있는 경우 | "이미 접수된 원격 건이 있습니다. 취소는 담당 선생님 또는 고객센터로 문의해 주세요." | [닫기] |
| 당일 종료된 접수 건이 있는 경우 | "원격 접수는 하루 1회만 가능합니다. 추가 문의는 담당 선생님 또는 고객센터로 연락해 주세요." | [닫기] |

### 페이지 3 — 공통 오류 ✅

예상치 못한 오류 또는 잘못된 접근 시 표시

```
┌──────────────────────────────┐
│  [서비스명] 배지               │
│  원격지원 서비스               │

├──────────────────────────────┤
│                              │
│  오류 제목                    │
│  안내 문구                    │
│                              │
└──────────────────────────────┘
```

**URL 파라미터**

| 파라미터 | 값 | 설명 |
|---------|-----|------|
| `type` | `token_expired` \| `invalid_access` \| `server_error` | 오류 유형 (기본값: `invalid_access`) |
| `service` | `SERVICE_MAP` 키값 | 서비스별 배지명·캐릭터 이미지 결정 (없으면 기본 이미지) |

**오류 유형별 메시지**

| 유형 (`type`) | 제목 | 안내 문구 |
|--------------|------|----------|
| `token_expired` | 링크가 만료되었습니다 | 원격 접수를 다시 요청하여 새 링크를 발송해주시길 부탁드립니다. |
| `invalid_access` | 잘못된 접근입니다 | 원격 접수를 다시 요청하여 새 링크를 발송해주시길 부탁드립니다. |
| `server_error` | 일시적인 오류가 발생했습니다 | 잠시 후 다시 시도해 주세요. |

**구현 포인트**
- 오류 유형에 따라 제목·안내 문구 교체 (`ERROR_MAP` 허용 목록으로만 제한)
- 확인 버튼 없이 메시지만 표시
- `service` 파라미터로 서비스명 배지 및 캐릭터 이미지 변경 (허용 목록 외 값 무시)
- 토큰 만료·잘못된 접근: 새 링크 발송 요청 유도 / 서버 오류: 재시도 안내

### 8-4. SP 호출 코드 예시 (PHP)

```php
function insert_remote_request(PDO $pdo, array $p): int {
    $sql = "SELECT remote_request_insert(
        :in_Reserv_Date,
        :in_Reserv_STime,
        :in_Reserv_ETime,
        :in_Service_Name,
        :in_Member_Status_Name,
        :in_reg_num,
        :in_user_id,
        :in_User_name,
        :in_Call_Tel,
        :in_Receiver_Emp_ID,
        :in_Reg_Date,
        :in_Remote_Count,
        :in_system_id
    ) AS result";

    $stmt = $pdo->prepare($sql);
    $stmt->execute([
        ':in_Reserv_Date'        => $p['reserv_date'],    // 'YYYYMMDD'
        ':in_Reserv_STime'       => $p['reserv_stime'],   // 'HHMM'
        ':in_Reserv_ETime'       => $p['reserv_etime'],   // 'HHMM'
        ':in_Service_Name'       => $p['service_name'],   // '밀크T초등'
        ':in_Member_Status_Name' => $p['member_status'],  // 세션에서
        ':in_reg_num'            => $p['reg_num'],        // 서버 생성
        ':in_user_id'            => $p['user_id'],        // 세션에서
        ':in_User_name'          => $p['user_name'],      // 세션에서
        ':in_Call_Tel'           => $p['phone'],          // 고객 입력
        ':in_Receiver_Emp_ID'    => $p['receiver_emp'],   // 세션에서
        ':in_Reg_Date'           => $p['reg_date'],       // 서버 생성 datetime
        ':in_Remote_Count'       => $p['remote_count'],   // 세션에서
        ':in_system_id'          => $p['system_id'],      // 세션에서
    ]);

    $row = $stmt->fetch(PDO::FETCH_ASSOC);
    return (int)($row['result'] ?? 0);  // 1=성공, 0=실패
}
```

## 10. 멀티스텝 폼 구현 상세

### 10-1. 개요

접수 폼을 5단계 스텝 위저드로 구성
연락처·문의내용·예약날짜·예약시간을 각 스텝에서 분리 입력 후, 마지막 스텝에서 전체 내용을 확인하고 접수

**구성 이유**
- 한 화면에 모든 필드를 나열할 경우 스크롤 없이 제출하거나 누락하는 사례 발생 → 단계 분리로 방지
- 각 단계에서 즉시 유효성 검사 → 오입력 즉시 인지 가능
- 날짜·시간을 한 화면에 함께 두면 모바일에서 시야가 좁아짐 → 날짜·시간을 각각 별도 스텝으로 분리
- 별도 모달 없이 마지막 스텝(Step 5) 자체를 확인 화면으로 사용하여, 화면 전환·포커스 이동을 최소화

---

### 10-2. 단계별 흐름

```
Step 1 — 연락처 입력
  └─ [다음] 클릭 시 연락처 유효성 검사
       ├── 실패: 에러 메시지 표시, 이동 불가
       └── 성공: Step 2로 이동

Step 2 — 문의내용 입력
  └─ [다음] 클릭 시 문의내용 유효성 검사
       ├── 실패: 에러 메시지 표시, 이동 불가
       └── 성공: Step 3으로 이동

Step 3 — 날짜 선택 (카드 그리드)
  └─ 날짜 카드 선택 시 자동으로 시간 슬롯 백그라운드 프리로드
  └─ [다음] 클릭 시 날짜 선택 여부 검사
       ├── 실패: 에러 메시지 표시, 이동 불가
       └── 성공: Step 4로 이동

Step 4 — 시간대 선택 (슬롯)
  └─ 슬롯 선택 시 시작·종료 시간 hidden input 반영
  └─ [다음] 클릭 시 시간 선택 여부 검사
       ├── 실패: 에러 메시지 표시, 이동 불가
       └── 성공: 입력값을 확인 패널에 채우고 Step 5로 이동

Step 5 — 입력 내용 확인 패널
  ├── [이전]: Step 4로 복귀 (입력값 유지)
  └── [접수하기]: AJAX 제출
         ├── 성공: 완료 페이지 이동
         └── 실패: 에러 배너 표시
```
